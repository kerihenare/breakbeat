Q7: How is the Haiku output schema actually enforced, and what exactly does Haiku return per result?

Enforcement options:

  • (a) Structured outputs — Anthropic's output_format: { type: "json_schema" } (beta, supported on Haiku 4.5). The API guarantees schema-conformant JSON via constrained decoding. No tool use, which honors the spec's prompt-injection stance, and the "validated output" claim becomes an API guarantee instead of app code.
  • (b) Prompt-requested JSON + JSON.parse + hand-rolled validation + one retry — the classic approach. More code, and the retry path is exactly the kind of edge that breaks during a live demo.

My recommendation: (a), with (b)'s validation kept as a thin sanity layer (check result IDs in the response match the IDs sent — schema can't enforce that). If the beta flag is unavailable in the SDK version, fall back to (b) — but it's GA-track and Haiku 4.5 supports it. Model pinned as a named constant: claude-haiku-4-5.

Per-result fields (the schema):

{ id, content_type: news|trade_publication|blog_post|press_release|social_post|newsletter|podcast|other,
  exclude: none|own_channel|ecommerce_review|aggregator,
  confidence: high|low }

Three judgement calls baked in there:

  1. exclude is an enum reusing the heuristic exclusion codes, so the LLM backstop writes the same exclusion_code vocabulary as the heuristics — one closed set across the funnel, as §1's soft-exclusion design demands. llm_excluded as a code disappears; instead exclusion_detail records "via LLM" — actually, simpler: keep the spec's codes but store the LLM's specific reason in exclusion_detail. I'd amend the spec: the code set stays closed, and LLM exclusions carry the real code (own_channel etc.) with detail "LLM", rather than a generic llm_excluded. Groupable by why, not by which stage caught it.
  2. One confidence flag covering both decisions (type + exclusion), not two — the UI only has one low-confidence treatment anyway.
  3. No free-text reasoning field. Tempting for debuggability, but it's the prompt-injection echo channel (hostile snippet text gets quoted back into your UI) and it burns output tokens × 400 results. The enums are the audit trail.

Q8: In name-only mode, what makes a homepage candidate "convincing"?

The spec says: name only → "use Tavily to find the homepage; if nothing convincing is found, proceed degraded." That word convincing is an unimplemented predicate, and it gates whether own-channel exclusion has any deterministic teeth for the entire job. Options:

  • (a) Trust Tavily rank 1 blindly — simple, but rank 1 for many company names is Wikipedia, LinkedIn, or Crunchbase, and the spec's degraded path exists precisely because this fails.
  • (b) Deterministic heuristic over the top ~5 results: take the first result whose domain is not on a "never-a-homepage" blocklist (wikipedia, linkedin, crunchbase, facebook, x.com, bloomberg, …) and whose domain or title contains a normalized token of the company name (acme ⊆ acme.com / getacme.io / title "Acme, Inc."). No match in top 5 → degraded.
  • (c) A small LLM call: give Haiku the top 5 results + company name, ask "which, if any, is the official homepage?" Most accurate, but adds an LLM dependency to a stage the spec treats as deterministic, and it's another injection-shaped surface ahead of the funnel.

My recommendation: (b). It's a pure function (unit-testable, in keeping with §3's testing scope), it fails closed — wrong-match risk collapses into the degraded path, which the spec already designs for, and the resolved identity is surfaced in the UI ("Identified as: acme.com") so a rare bad match is visible immediately. The name-token check is the key guard: it converts "Tavily's best guess" into "Tavily's best guess that's at least nominally plausible."

One sharp edge worth naming: companies whose domain shares no token with their name (e.g. Alphabet → abc.xyz). (b) goes degraded there — which is the correct failure: degraded mode warns and lets the LLM backstop carry own-channel, rather than confidently excluding the wrong company's domain. And the form already hints "provide the URL for unambiguous results."

(b) → (c) cascade:

Q10: How exactly is the results list organized — section order, sort-within-group, and where do the awkward rows go?

The spec says "grouped/filterable by content type" but leaves the layout decisions open. There are three awkward row classes that need homes: date-unknown results, unclassified results (classify-stage failure → nullable type), and the excluded section. My proposal:

  1. Fixed section order by editorial weight, not alphabet: News → Trade publications → Press releases → Podcasts → Blog posts → Newsletters → Social posts → Other → Unclassified. Rationale: an evaluator scans top-down; the highest-signal third-party coverage should hit them first. Each section header carries a count, and empty sections are omitted entirely (an empty "Newsletters (0)" is noise).
  2. Within a section: published date descending, date-unknown rows last with a visible "date unknown" badge. Recency-first is how humans review coverage; unknowns sinking to the bottom keeps the dated timeline clean while keeping recall.
  3. Unclassified is its own section, only present on classify failure, labeled honestly ("Unclassified — classification failed, see warnings") rather than dumped into Other. The spec already insists other means genuine type ambiguity, never a failure default — the UI should preserve that distinction.
  4. Excluded: one collapsed <details> block at the very bottom, grouped by exclusion_code with counts ("own_channel (12) · duplicate (9) · aggregator (5)"), each row showing its exclusion_detail. Collapsed by default — it's the audit trail, not the product.
  5. Low-confidence flag: an inline badge on the row, not a separate section — a low-confidence news item is still news; segregating it would break the by-type review flow.
  6. Filtering (slice 5): simple anchor links / a type chip row that jumps to or shows-hides sections. No search box, no client templating — just visibility toggles. This is the first thing cut under time pressure per §4, and the fixed grouping already delivers "reviewable."

Agree with all six — particularly the editorial section order in #1 over the brief's listing order (news articles, trade pubs, blogs, press releases, social, newsletters, podcasts)?
