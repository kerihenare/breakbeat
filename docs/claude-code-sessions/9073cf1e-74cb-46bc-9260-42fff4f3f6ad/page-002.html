Q3: Where does the SQLite database live, and what's its lifecycle?

The spec picks node:sqlite but never says where the file goes or how the schema comes to exist. Sub-decisions:

  1. File location — ./data/breakbeat.db (gitignored, directory created at boot) vs project root vs :memory:. Memory is out — the spec explicitly wants durability ("jobs are durable rows", "durable enough to demo dedup properly").
  2. Schema creation — CREATE TABLE IF NOT EXISTS idempotently at boot vs a migrations system. For a local-only MVP with one author, migrations are ceremony; if the schema changes during development, you delete the file.
  3. Reset story — does a cloner/evaluator ever need a fresh DB? A db:reset npm script (delete file) is two lines and makes the demo repeatable.
  4. Pragmas — journal_mode = WAL and foreign_keys = ON. WAL matters here more than it looks: the HTMX poll reads while the pipeline writes, and default rollback-journal mode can hit SQLITE_BUSY on that overlap.

My recommendation: ./data/breakbeat.db, gitignored with the data/ dir; idempotent CREATE TABLE IF NOT EXISTS schema in db.ts run at boot; a db:reset script; WAL + foreign keys on. No migrations — note it as deliberate in the write-up (the schema ships final; evolution is a hosted-version concern).

Q4: What exactly does "normalized URL" mean?

This is a pure function the spec promises unit tests for, so let's spec it. My proposed rules, with the contentious ones flagged:

  1. Lowercase host, strip www. — uncontroversial.
  2. Drop the scheme from the key (store the original URL separately for the link) — so http:// and https:// copies of the same article collapse. The displayed link always uses the original URL.
  3. Strip fragment (#...) — never identifies distinct content. Uncontroversial.
  4. Strip only known tracking params — utm_*, fbclid, gclid, mc_cid, ref, source — and keep everything else. This is the trap: stripping all query params feels cleaner but merges genuinely distinct content (youtube.com/watch?v=abc vs ?v=xyz, ?id=-style article URLs). Allowlist-of-trackers-to-remove, not allowlist-of-params-to-keep.
  5. Sort remaining query params — so param order doesn't defeat dedup.
  6. Strip trailing slash on path; preserve path case — paths are case-sensitive in the wild; hosts aren't.
  7. Uniqueness scope: per job — UNIQUE(job_id, normalized_url), matching the spec's "per job" parenthetical. Cross-job dedup would be wrong: two different jobs (even for the same company) should each show full results.

My recommendation: all seven as stated. The one I'd flag as a judgement call is #4's ref/source — some sites use ?source= meaningfully. I'd still strip them: in this corpus (news/blog URLs from a search API) they're overwhelmingly trackers, and a rare false merge of two URLs differing only by ?source= is almost certainly the same article anyway.

Next branch: the other half of dedup — title Collapse. The spec pins the guards (≥25 chars, ±14 days, earliest-date wins) but "normalized title" is just as undefined as "normalized URL" was, and it's doing more subtle work.

Q5: What does title normalization do — and specifically, does it strip outlet suffixes?

The base rules are easy and I'll assert them rather than ask: Unicode NFKC normalize, lowercase, strip punctuation, collapse whitespace, then exact match on the result (the spec says "normalized title", not fuzzy similarity — embeddings are explicitly out of scope in §5).

The real decision is the suffix problem. Syndicated copies usually differ only by outlet branding:

  • Acme raises $50M Series B | TechCrunch
  • Acme raises $50M Series B - Yahoo Finance

Without suffix stripping, the exact-match collapse misses precisely the syndication case it exists to catch. Options:

  • (a) No suffix stripping — simple, but defeats the stage's main purpose for the most common dup pattern.
  • (b) Strip a trailing | X / – X / - X segment, with guards: only the final separator, only if the remaining title is still ≥25 chars (so Acme - The Real Story doesn't become Acme), and only if the stripped segment is short (≤~40 chars — outlet names are short; a real subtitle usually isn't).
  • (c) Strip the segment only when it matches the source domain (e.g. suffix "TechCrunch" on a techcrunch.com URL). Most precise, but then the syndicated copy on yahoo.com saying "| TechCrunch" doesn't get stripped — which is backwards: the syndicated copy is exactly the one you need to normalize.

My recommendation: (b). The 25-char remainder guard already exists in the spec's collapse rules, so this composes cleanly; the ±14-day window catches any residual false merges. And it's a pure function — easy to unit test with exactly these adversarial cases (Acme - The Real Story, the TechCrunch/Yahoo pair).

Approve (b) with those two guards, or do you see a title shape it mangles?

Q6: Which Tavily topic and search_depth does each query group use?

Tavily's API shapes results differently per topic:

  • topic: "news" — queries a news-specific index, returns reliable published_date, and supports start_date/end_date. The spec's time-slicing depends on this ("the two types where Tavily dates are reliable" — that reliability comes from the news topic).
  • topic: "general" — broader web index, dates often missing (hence the spec's "date unknown" flag), no trustworthy date filtering.

And search_depth: "advanced" returns better-ranked results with richer snippets but costs 2 credits vs 1 per query — at 18 queries/job that's 36 vs 18 credits (free tier: 1,000/month, so ~27 jobs vs ~55).

My recommendation: 1. News-ish queries (the "X news" + "X press release" type queries, and all 6 time-sliced queries) → topic: "news". The date filter is real there. 2. Everything else (blogs, podcasts, social, newsletters, trade pubs, the 5 angle queries) → topic: "general" with time_range: "3y" as a best-effort filter — accepting the spec's position that dateless results are kept and flagged. 3. search_depth: "advanced" everywhere. This is the one place I'd spend. The spec names result quality "the deciding factor in every trade-off," and classification runs on title+snippet only (§5 — no page fetching) — so snippet quality is classification quality. Richer advanced-mode snippets feed directly into Haiku's accuracy. 36 credits/job is still ~27 free jobs — plenty for a demo. Name it as a deliberate cost trade-off in the write-up, with search_depth as one of the named constants so it's a one-line downgrade.
