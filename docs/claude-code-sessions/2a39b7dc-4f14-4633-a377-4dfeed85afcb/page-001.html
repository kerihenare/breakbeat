Read both the spec and the exercise brief. Overall assessment: strong spec, well-calibrated to the 4-hour box — but there are a few gaps worth fixing before implementation, one of which is a genuine requirements miss.

What's good

  • Trade-off table maps directly onto the evaluation criteria (cost awareness, dedup, background job design, retrieval strategy). The "chose X over Y because" framing is exactly what the write-up deliverable wants — much of section 2 can be lifted straight into it.
  • Tracer-bullet slicing is the right call: a working stub loop after slice 1 means you always have something demoable, which de-risks the hard time cap.
  • Sensible boring tech: node:sqlite (zero deps for the cloner), in-process queue with a portable state-machine interface, polling over SSE. All defensible.
  • Partial-failure handling (done_with_warnings, keep unclassified results) and "flag low-confidence rather than silently drop" show good product judgement for a reviewable list.
  • Out-of-scope section is explicit and justified — handles the ambiguity criterion head-on.

Gaps and risks, in priority order

  1. Own social channels aren't excluded — requirements miss. The brief excludes "their own website, blog, LinkedIn, etc." Your heuristics only drop own domains/subdomains (docs/1-spec.md:27). The company's own linkedin.com/company/x page or x.com/<handle> posts won't match a domain rule — and you can't blanket-block social domains because "major social posts" about the company are explicitly in-scope. You need: (a) capture social handles during Resolve (often in homepage metadata/footer), and/or (b) an explicit instruction in the Haiku classify pass: "exclude posts authored by the company; include posts about it."

  2. Missing publish dates are unhandled. A large share of search results have no reliable date. The heuristic drops "out-of-window dates" — but is undated content dropped or kept? Silently dropping would gut recall; silently keeping breaks the 36-month promise. Recommend: keep, flagged "date unknown", consistent with your low-confidence philosophy. One line in the spec saves an ambiguous decision mid-build. (Also verify Tavily's start_date/end_date actually constrain publish date — its date filtering is historically unreliable, which makes this case common, not edge.)

  3. The plan sums to exactly 240 minutes — zero buffer. Something will slip (usually slice 2, query tuning). Mark slices 5 (UI polish) and part of 4 as the sacrificial buffer explicitly, and consider doing slice 6's README skeleton inside slice 1 so a time-out still leaves a clone-and-run repo. The deliverables (README, write-up, transcript) are non-negotiable; UI polish isn't.

  4. TypeScript runtime is unstated. The tree shows .ts files and the persistence choice requires Node 22+, but nothing says how TS runs — tsx, or --experimental-strip-types? This directly affects the "clone → .env → run" criterion. Pin a Node version and runner in the spec now (e.g. Node 22.6+ with type stripping, or just tsx as the one dev dep).

  5. POST / returning the view breaks refresh semantics. Redirect to GET /:id (PRG) instead — it's also free progress UX since that's already the polling target. Relatedly, say whether polling returns JSON or HTML fragments; ~2s full-page HTML polling is fine but decide once.

  6. Minor: the single batched Haiku call needs a chunking fallback if surviving results exceed a comfortable output size (your "max results to classify" cap mostly covers this — just make the cap the chunk size). And when name-only input resolves a homepage, surface "Identified as: acme.com" in the job status — cheap mitigation for the disambiguation limitation you already acknowledge.

Verdict

Ship it after addressing #1 and #2 (both are one-or-two-line spec edits), and restructure the time budget per #3. The architecture itself needs no changes — it's appropriately minimal, and the upgrade paths (BullMQ, multi-provider, embeddings dedup) are correctly identified as narrative for the write-up rather than work.
