I've read the spec, the brief, CONTEXT.md, and the current repo state. Here's my assessment.

Overall verdict

This is a strong spec — notably well-aligned with the brief's grading rubric. Nearly every evaluation criterion has a deliberate, named answer: soft exclusion makes "result quality and deduplication" visible and debuggable, the trade-off table addresses "sensible trade-offs" head-on, the budget caps and SSRF guard cover "security and cost awareness", and the named-and-skipped items (DNS rebinding, queue durability, semantic dedup) show judgement rather than omission. The tracer-bullet slicing is the right call for a 4-hour window: a working end-to-end loop after slice 1 means you always have something demoable.

The main risk isn't design quality — it's scope vs. the 4-hour budget.

Strengths worth keeping

  • Soft exclusion with reasons (§1.3, line 35) — the single best product decision in the spec. It turns the filtering strategy into a reviewable artifact, which is exactly what an evaluator will poke at.
  • Time-slicing only dated types (line 30) — the reasoning ("paying 3× for noise" on dateless types) is correct and shows retrieval thinking, not just query spam.
  • Collapse guards (line 34) — title-length minimum and ±14-day window prevent the classic false-collapse failure ("same headline 18 months apart is two stories").
  • The homepage fetch as cost optimisation, not guarantee (line 27) — correctly identifies the LLM as the own-channel backstop. The brief's hardest exclusion ("their LinkedIn") is properly handled by the authored-by vs. about distinction.
  • Partial-failure path (done_with_warnings, line 81) — keeping unclassified results beats failing the job.

Concerns

1. Scope risk (the big one). Slices 1–6 plus unit tests, SSRF guard with per-hop redirect re-checking, social handle scraping, time-sliced queries, progressive rendering, and confidence flags is a lot, even agent-assisted. If time runs short, the spec doesn't say what gets cut. I'd add an explicit de-scoping order — e.g. cut first: social handle scraping (LLM backstop already covers it), time slicing (angle queries carry recall), per-hop redirect re-check (cap redirects at 0 instead). Knowing your cuts in advance is itself a graded "sensible trade-off."

2. "Major" social posts — unhandled ambiguity. The brief says major social posts; the spec never addresses what "major" means or that Tavily gives you no engagement signal. Since "handling ambiguity" is an explicit criterion, this deserves one line — even just "we interpret 'major' as 'surfaced by search ranking'; engagement-based filtering needs native social APIs (out of scope)."

3. Prompt injection in the classify pass. Line 75 covers escaping untrusted Tavily content on render, but that same untrusted content also lands in the Haiku prompt. The blast radius is tiny (output is a constrained classification, validated), but a Head-of-Engineering reviewer may look for it to be named. One sentence: "snippet content is untrusted prompt input; mitigated by constrained/validated output schema and no tool use in the classify call."

4. Cost estimate looks overstated. ~18 Tavily queries is roughly 18–36 credits ≈ $0.15–0.30/job at pay-as-you-go rates, plus negligible Haiku spend. "$1–2/job" errs safe, but in a graded exercise an inflated estimate slightly undercuts the cost-awareness signal. Worth a quick re-check against current Tavily pricing.

5. Minor internal inconsistency. The architecture comment (line 58) gives the state machine as …→done|failed, while §3's canonical version (line 68) includes done_with_warnings. Trivial, but it's the kind of thing a careful reviewer notices in a spec that's otherwise this tight.

6. Node type-stripping constraint is unstated. "No build step" via Node 26 native TS is fine, but it means erasable syntax only — no enum, no namespace, no constructor parameter properties. Worth one line so it doesn't bite mid-build (an agent will happily emit an enum).

7. Whole-body swap every 2s loses scroll position and any client-side filter state mid-job. Acceptable (results are mostly reviewed after completion), but the spec lists "a result can vanish" as the trade-off and misses this second one. Cosmetic; mention or ignore.

Small product question

Boot requires both keys, but classify failure already degrades gracefully to unclassified results. Allowing boot with only TAVILY_API_KEY (warn + skip classify) would make the clone-and-run experience more forgiving for an evaluator who only sets up one key. Cheap to do, optional.

Bottom line

The spec would I think score well as-is — it reads like someone who has built media-monitoring pipelines before. The actionable fixes are: add a de-scoping order (#1), one line each for "major" social ambiguity (#2) and classify-prompt injection (#3), and sanity-check the cost figure (#4). Everything else is polish.
