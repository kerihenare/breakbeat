## Task

Build a basic web app that accepts:

- A company name; and/or

Content types may include:

- News articles

Content should not include:

The user sees "Identified as: apexlegends.com"... and then what?

Options:

a) Nothing — fully automatic. Wrong match = start a new job with the URL this time. Wasted credits are pennies. b) Confirmation gate — job pauses at resolving, user confirms the identity before search spends anything. My recommendation: (a). A confirmation gate adds a state, a form, and an interaction mode to a 4-hour build, and it punishes the 90% case (correct match) with friction. The wasted-cost downside is bounded by your per-job budget cap (~10–15 Tavily credits ≈ cents). The mitigation is already in the spec: show the resolved identity prominently and early. The write-up then names the confirmation gate as the obvious next step for handling ambiguity — which also scores points on the "handling ambiguity" criterion without building it.

The one thing I'd add to the spec either way: the form should say something like "tip: provide the URL for unambiguous results" — product judgement on display for free.

Question 2: The homepage scrape is your source for "own channels" — but what happens when it returns nothing useful?

Two failure scenarios that hit real companies constantly:

  1. JS-rendered SPAs. You fetch acme.com, get a React shell — no footer links, no LinkedIn URL, maybe not even a meaningful <title>. Your social-handle scrape comes back empty, so own-channel exclusion is running on the homepage domain alone. Then linkedin.com/company/acme sails through the heuristic filter.
  2. Multiple own domains. The company is acme.com but their blog is blog.acme.io, their press releases are on acmecorp.com. Subdomain matching catches blog.acme.com, not sibling domains.

The spec treats the scrape as if it works. Options:

a) Best-effort + LLM backstop. Regex the raw HTML for social profile URLs; whatever you find improves the heuristic pass. Misses fall through to the Haiku classify stage, which is already tasked with detecting "content authored by the company on third-party platforms." Accept sibling-domain leakage as a known limitation. b) Heuristic-grade guessing. Also probe common patterns (linkedin.com/company/{name}, x.com/{name}) and treat near-matches of the company name in social URLs as own-channel. c) Headless browser for proper rendering. (Obviously no — dependency weight, time.)

My recommendation: (a), and make it explicit in the spec. The key reframe: the scrape isn't your own-channel defence, it's a cost optimisation — every own-channel item it catches heuristically is one fewer item in the Haiku batch. The classify prompt is the actual guarantee, and it needs to handle this anyway (the scrape can never be exhaustive — TikTok, YouTube, Substack...). (b) sounds clever but generates false positives (x.com/apex might be someone else entirely) for marginal gain.

One concrete addition this implies: the classify prompt should receive the resolved company name + domain + any scraped handles as context, so Haiku can judge "authored-by vs about" properly. Is that already your mental model, or were you counting on the scrape as the primary exclusion mechanism?
