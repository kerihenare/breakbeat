Question 11: The homepage fetch — how hardened, exactly?

The spec says "SSRF guard: reject private IPs," which is the right instinct but underspecified, and this fetch is the only place the app touches a user-controlled URL. Threat model honestly: local-only app, single user, attacking yourself — SSRF here is about demonstrating awareness, not real exposure. So the question is which guards make the demonstration without eating your time box:

a) DNS-check only. Resolve hostname, reject private/loopback/link-local ranges, fetch. b) (a) + fetch hygiene: redirect cap (~3, re-checking each hop's host — bit.ly → 169.254.169.254 is the classic), timeout (~5s), response size cap (~1MB), https?: schemes only, and re-validate after redirects. c) Full paranoia: DNS-rebinding-proof (pin the resolved IP for the actual connection), IPv6 edge cases, etc.

My recommendation: (b). (a) alone has a hole you'd be embarrassed about in review (redirect to internal address after the check passes). (c) is genuinely hard in Node without a custom agent and is production-grade effort the brief explicitly doesn't want. (b) is ~25 lines, and the redirect re-check is the line that shows you actually understand the attack rather than pattern-matching "block 10.x".

Two adjacent things to fold into the same answer:

  • Timeout/size caps aren't only security — they're what stops a slow homepage from hanging the Resolve stage and, with your FIFO-1 queue, every job behind it. That's the stronger reason they're non-optional.
