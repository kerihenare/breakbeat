Question 10: While a job runs, does the user watch a status line — or results filling in?

The spec's polling mechanism (page re-fetches itself every ~2s, swaps the body) is settled and right. But it leaves open what's on the page mid-job. The pipeline produces results incrementally: after Search, rows exist; Filter and Classify then mutate them. Scenario: 40 results land at searching, then over the next ~20s some flip to excluded and types appear. Do I see that happen?

Options:

a) Status only until terminal. Stage indicator while running; results render only at done. Simple, but the user stares at "classifying…" for 30s with nothing to read. b) Progressive render. Status header + whatever results currently exist, in whatever state they're in. The 2s swap means the list visibly changes — items appearing, then getting excluded, types appearing. Free with your architecture (the page is server-rendered from the DB each poll anyway). c) (b) + per-stage counters in the status header ("47 fetched · 12 excluded · 31 classified").

My recommendation: (c), where (c) is maybe ten lines over (b). Two reasons. First, the demo: a list that visibly assembles itself is your background-job story told in the UI — the evaluator watches the pipeline work instead of reading about it. Second, honesty about your own mechanism: since the body swap re-renders from the DB regardless, (a) would actually mean suppressing data you already have. The counters in (c) come from one GROUP BY query and make stage progress legible even when individual rows are hard to eyeball.

One caveat to accept with eyes open: mid-job, a result the user is reading can vanish into the excluded section on the next swap. At a 2s cadence with ~30–60s jobs, that's a feature (the filter visibly working), not a bug — but it's the trade-off (b)/(c) makes.

Also confirming the terminal behaviour while we're here: the swap loop stops when the rendered page carries a terminal state (done / failed / done_with_warnings) — no polling forever on finished jobs.

Question 11: The homepage fetch — how hardened, exactly?

The spec says "SSRF guard: reject private IPs," which is the right instinct but underspecified, and this fetch is the only place the app touches a user-controlled URL. Threat model honestly: local-only app, single user, attacking yourself — SSRF here is about demonstrating awareness, not real exposure. So the question is which guards make the demonstration without eating your time box:

a) DNS-check only. Resolve hostname, reject private/loopback/link-local ranges, fetch. b) (a) + fetch hygiene: redirect cap (~3, re-checking each hop's host — bit.ly → 169.254.169.254 is the classic), timeout (~5s), response size cap (~1MB), https?: schemes only, and re-validate after redirects. c) Full paranoia: DNS-rebinding-proof (pin the resolved IP for the actual connection), IPv6 edge cases, etc.

My recommendation: (b). (a) alone has a hole you'd be embarrassed about in review (redirect to internal address after the check passes). (c) is genuinely hard in Node without a custom agent and is production-grade effort the brief explicitly doesn't want. (b) is ~25 lines, and the redirect re-check is the line that shows you actually understand the attack rather than pattern-matching "block 10.x".

Two adjacent things to fold into the same answer:

  • Timeout/size caps aren't only security — they're what stops a slow homepage from hanging the Resolve stage and, with your FIFO-1 queue, every job behind it. That's the stronger reason they're non-optional.
  • Input validation scope: company name → length cap + non-empty, nothing clever; URL → URL parse + scheme check, then the SSRF guard does the rest. Anything more is theatre — it's all rendered server-side into HTML (escape on output, the actual XSS defence, since search results are also untrusted strings you'll render).

That last point is worth pinning: result titles/snippets from Tavily are untrusted input too — a malicious page title containing <script> lands in your HTML. Escaping at render is the one security control the spec doesn't mention and definitely needs.

Agree with (b) + output escaping as stated?

Question 12: The budget caps need numbers — what are they?

The spec says "max queries, max results to classify" but never commits. These numbers shape the Haiku batching, the latency, and the cost claim in your write-up, so they should be in the spec, not invented mid-implementation. Working through what we've already decided:

  • Max queries/job: we settled ~15 (7 content types + 4 news/PR time slices + 4 angles). Cap at 16 — the strategy is the cap; a hard ceiling above it is just a guard against future drift.
  • Results per query: Tavily max_results — recommend 10 (the default 5 is thin for a recall exercise; 20 doubles noise for tail-quality hits). 15 queries × 10 ≈ 150 raw results pre-dedup.
  • Max results to classify: post-heuristics, realistically 60–100 survive. Recommend cap 150, chunk size 50 (spec already says cap = chunk size, so: ≤3 Haiku calls/job). A 50-item batch of title+snippet+URL is ~10k input tokens — well inside Haiku's comfort zone, and chunking bounds the blast radius of one malformed response to a third of the results.
  • What happens at the cap: consistent with soft exclusion — results past 150 aren't dropped, they're kept unclassified and flagged, same as your classify-failure path. The cap throttles spend, never recall.

Cost picture this produces per job: ~15 Tavily credits + ~30k Haiku tokens ≈ $0.15–0.75 depending on Tavily plan, with LLM cost a rounding error — a concrete number for the write-up, which beats "cost-aware" as an adjective.

My recommendation is those four numbers as stated, as named constants in one place (config.ts or top of pipeline.ts) rather than scattered literals — they're the knobs the write-up's "next steps" will say become per-plan quotas.

Question 13 (last one): the repo's plumbing doesn't match the spec — fix as follows?

Cross-referencing code against everything we've agreed:

  1. Broken entrypoint: package.json says "main": "./src/index.ts" but the file is src/main.ts (which is also what the spec's architecture names). pnpm dev (node .) fails today. Fix: point main at ./src/main.ts, or just make dev explicit.
  2. .env loading with zero deps: Node 26 runs TypeScript natively and loads env files natively — so "dev": "node --env-file=.env --watch src/main.ts" gives you env keys + auto-reload with no dotenv, no tsx, no nodemon. This matters beyond tidiness: "clone → add keys → run" is graded, and a missing --env-file is the classic way that flow breaks. Caveat to handle: --env-file errors if .env is absent — use --env-file-if-exists=.env so the app can boot and show a friendly "missing TAVILY_API_KEY" message instead of a launch error.
  3. .env.example: exactly two keys, TAVILY_API_KEY and ANTHROPIC_API_KEY, each with a one-line comment linking where to get it. Boot-time check prints which are missing and exits cleanly.
  4. Express stays the sole runtime dependency — given the zero-dep aesthetic everywhere else (node:sqlite, node:test, native env/TS) you could go all the way with node:http, but hand-rolled routing/parsing is undifferentiated code that eats minutes and impresses nobody. One boring dependency is the right trade-off; the restraint elsewhere already makes the point. (Anthropic + Tavily calls via plain fetch — no SDKs needed for one endpoint each, consistent with the same ethos.)

Recommendation: all four as stated. These are small, but #2's --env-file-if-exists + friendly key check is genuinely load-bearing for the graded clone-and-run experience.
